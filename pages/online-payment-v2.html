Online Payment API V2

Overview

Online Payment API V2 is the dual-auth payment platform for MITF (.NET 10, FastEndpoints, PostgreSQL). One host serves legacy JWT clients (proxied verbatim to frozen V1) and new API-key integrators on RESTful /v2/*. V2 owns payment sessions, OTP, refunds, and inquiries; bank settlement goes through Legacy Core over gRPC.

Key Features

  • Dual auth — JWT (V1 shape) and x-api-key (mitf_live_* / mitf_test_*) on one host
  • Payment sessions — open, complete (OTP verify), refund; state in PostgreSQL
  • Inquiries — balance, transactions, status, account reports
  • JWT proxy — /api/OnlinePaymentServices/* forwarded to V1 unchanged
  • Control plane — idempotency, rate limits, quotas, audit logging
  • RFC 7807 errors — stable code on /v2/*; Swagger at /swagger

Capabilities

Payment flow: Open session → OTP to customer → verify hash → Legacy Core Pay → refund via session if needed.

Operations: Redis-cached key resolution (30s), Idempotency-Key retries, per-merchant quotas, full mutation audit.

Security: API keys resolved via portal resolve-key (never stored in V2); OTP hashed before bank call; no direct V1 bank client access.

Main /v2/* routes: POST /payment-sessions, POST .../complete, POST .../refund, GET /balance, GET /transactions, GET /reports/account.

Business Value

Audience Value
Merchants REST payments without bank creds; sandbox keys; quota protection
Customers OTP confirmation; familiar flow; idempotent retries
Administrators Audit trail, rate limits, single endpoint for all traffic
Partners Documented REST surface, prefixed keys, machine-readable errors

Highlights

  • Zero-disruption migration — JWT clients work through V2 proxy; V1 contracts untouched
  • Production controls built in — idempotency, quotas, audit at the API layer
  • Bank isolation — Legacy Core is the sole bank touchpoint

Summary

V2 is the execution layer for MITF online payments: legacy and modern auth on one host, full session lifecycle, and enterprise controls. Partners use API keys; bank credentials stay in the portal vault.
